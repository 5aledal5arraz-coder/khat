# Database migrations

Versioned, forward-only SQL migrations generated by `drizzle-kit`. This replaces
`drizzle-kit push` for shared databases (local-shared and production).

**Why not `db:push`?** `push` diffs the schema against the live DB and, for any
ambiguous change, **stops to ask interactively** "is this table/column *created*
or *renamed*?" — un-automatable, and a wrong answer renames a table and loses
data. Versioned migrations are explicit committed SQL: deterministic,
reviewable, forward-only, no guessing. `db:push` is now **local-throwaway only**.

## Files

- `0000_baseline_full_schema.sql` — the baseline: the entire schema as of
  adoption. Generated with `drizzle-kit generate`; verified to build all 108
  tables from scratch on a clean database.
- `meta/_journal.json`, `meta/*_snapshot.json` — drizzle's migration journal and
  schema snapshots. **Commit these.** They are how `generate` diffs and how
  `migrate` tracks order. Do not hand-edit.
- `../_pre-baseline-archive/` — the pre-baseline hand-written migration, kept for
  history. Its columns are already included in the baseline; see that folder's
  note.

> `scripts/post-schema.sql` is still applied **after** migrations. It holds
> triggers, RPC functions (`push_episode_data`), CHECK constraints, and a few
> indexes that aren't modeled in the Drizzle schema (e.g. `uq_gdl_candidate`).
> Migrations build the tables; post-schema layers the rest.

## Day-to-day: changing the schema

1. Edit the Drizzle schema under `lib/db/schema/`.
2. `npm run db:generate` — writes a new `NNNN_<name>.sql` + updates the journal.
   For an ambiguous rename it asks **once, at author time** (reviewable, in dev) —
   never against a live DB.
3. **Read the generated SQL.** It's committed; treat it like code.
4. `npm run db:migrate` — applies pending migrations locally.
5. Commit the `.sql` + `meta/` changes together.
6. Deploy: run `npm run db:migrate` against production, then re-apply
   `scripts/post-schema.sql`.

## One-time: adopting an EXISTING database

Production (and any local DB already at the schema) already *has* these tables,
so running `migrate` blind would fail trying to re-create them. Baseline it once:

```bash
DATABASE_URL="postgres://…prod…" npm run db:adopt
```

`db:adopt` (`scripts/adopt-migrations.ts`) records every journaled migration as
"already applied" — writing the exact rows `drizzle-kit migrate` writes
(verified: `hash = sha256(.sql file)`, `created_at = journal "when"`). It refuses
to run unless the `episodes` table exists (so it can't be misused on an empty
DB), never creates app tables, and is idempotent. Afterward, `db:migrate` only
applies migrations newer than the baseline.

## A fresh/empty database

```bash
DATABASE_URL="postgres://…new…" npm run db:migrate   # builds the full schema
DATABASE_URL="postgres://…new…" npx tsx scripts/run-migration.ts scripts/post-schema.sql
```

No `db:adopt` needed — `migrate` runs the baseline because nothing is recorded yet.
